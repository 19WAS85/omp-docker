# Repository Guidelines

## Project Overview

Oh My Pi (OMP) is a Dockerized development harness for an AI-powered coding agent. The repository contains **no application source code** — it is purely the infrastructure layer that builds, installs, and runs the OMP agent inside a container. The agent itself (`@oh-my-pi/pi-coding-agent`, `@oh-my-pi/pi-natives`) is installed as an npm package at image build time.

## Architecture & Data Flow

```
Host machine                    Docker container
─────────────                   ────────────────
run.sh ──docker compose──►      entrypoint.sh
build.sh ──docker compose──►      ├─ git config (safe.directory, user email/name)
install.sh (symlinks only)       ├─ omp update
                                  └─ exec omp (or passthrough to bash/sh)
                                 /work ←── mount of host $PWD
                                 /root/.omp ←── mount of host ~/.omp (agent state)
```

**Lifecycle:**
1. `install.sh` builds the Docker image and creates `~/.local/bin/omp` → `run.sh` symlink
2. `run.sh` invokes `docker compose run --rm omp` with the host CWD mounted at `/work`
3. `entrypoint.sh` configures git, runs `omp update`, then `exec omp` (or passthrough commands)

**Key design decision:** The container receives `NET_ADMIN` + `NET_RAW` capabilities for network-level operations the agent may need.

## Key Directories

| Path | Purpose |
|------|---------|
| `docs/superpowers/plans/` | Implementation plans generated by subagents |
| `docs/superpowers/specs/` | Design specs for features |
| `/work` (in container) | Host project workspace, bind-mounted |
| `/root/.omp` (in container) | Agent state directory, bind-mounted from host `~/.omp` |

This repository has **no `src/`, `lib/`, `app/`, or `test/` directories**. All executable logic is in 4 shell scripts at the root.

## Development Commands

```bash
# Build the Docker image
./build.sh
# or: docker compose build

# Run the agent interactively in the current directory
./run.sh
# or: docker compose run --rm omp

# Run a specific command inside the container
./run.sh bash
./run.sh python3 script.py

# Full install (build + create ~/.local/bin/omp symlink)
./install.sh
```

## Important Files

| File | Purpose |
|------|---------|
| `Dockerfile` | Image definition: oven/bun base, system deps, Bun global packages, Rust NAPI tokenizer addon |
| `compose.yaml` | Single-service Docker Compose config (service: `omp`) |
| `entrypoint.sh` | Container entrypoint: git config → omp update → exec omp |
| `run.sh` | Host-side orchestrator: resolves paths, sets OMP_WORKDIR, runs docker compose |
| `build.sh` | Thin wrapper around `docker compose build` |
| `install.sh` | Builds image + creates `~/.local/bin/omp` symlink |
| `.gitignore` | Ignores `.last-build` (checksum cache from original smart-build design) |

## Runtime & Tooling Preferences

- **Container base:** `oven/bun` (Debian-based with Bun pre-installed)
- **Host runtime:** Bash only. No package manager, no linter, no formatter on the host side
- **Inside container:** Bun (JS runtime), Python 3 with venv + ipykernel, ripgrep, fd-find, Rust toolchain (for building native addons)
- **Global npm packages:** `@oh-my-pi/pi-coding-agent`, `@oh-my-pi/pi-natives`
- **Native addon:** `@anush008/tokenizers` — built from source via `cargo build --release` for `aarch64-unknown-linux-gnu`
- **No Makefile, no package.json, no tsconfig at project root** — this is not a JS/TS project

## Code Conventions & Common Patterns

### Shell Scripts

All scripts use strict mode and deterministic path resolution:

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$(readlink -f "$0")")" && pwd)"
```

- **`set -euo pipefail`** — mandatory in every script (strict mode: exit on error, undefined vars, pipe failures)
- **`readlink -f`** for canonical path resolution (GNU extension, acceptable since target is Debian)
- **`$(cd "$(dirname ...")" && pwd)`** for reliable SCRIPT_DIR
- **`exec`** for clean process replacement (no orphaned shells)

### Docker

- Single service (`omp`) in `compose.yaml` — no multi-service orchestration
- Host directories mounted via environment variable defaults (`${OMP_WORKDIR:-.}`)
- Capabilities added explicitly (`NET_ADMIN`, `NET_RAW`)

## Testing & QA

There is **no test infrastructure** in this repository. No test frameworks, no test files, no CI/CD pipelines. The shell scripts use `set -euo pipefail` as the primary safety mechanism. Verification is manual: build the image and run the agent.

## Documentation

Design specs and implementation plans live under `docs/superpowers/` and are generated by subagents during development.
